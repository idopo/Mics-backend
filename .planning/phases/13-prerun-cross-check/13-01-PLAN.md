## Implementation

### Step 1: Server-side preflight validate endpoint

**File:** `api/routers/toolkits.py`

```
POST /api/sessions/{session_id}/preflight-validate/{pilot_id}
```

The UI cannot reliably know the `task_definition_id` for a session's protocol step — this must be resolved server-side. This endpoint resolves the task_definition_id internally from the session's protocol step:

Logic:
1. Load session → get `protocol_id` and current step
2. Load protocol step → get `task_definition_id` (if present)
3. If no `task_definition_id` or toolkit is not backend-authored: return `{ok: true, issues: [], skip_reason: "not_backend_authored"}`
4. Load task definition → get `toolkit_id`
5. Load toolkit → get `hardware_module_ids`
6. For each `hardware_module_id`:
   a. Load hardware module → get `name`, `class_name`, `hardware_lib_id`
   b. Check `pilot_hardware_config` for `(pilot_id, hardware_module_id)` row
   c. If no row: issue `{module_name, issue: "missing", detail: "..."}`
   d. If row exists but config is empty: issue `{module_name, issue: "incomplete_config", detail: "..."}`
   e. If row's stored class_name differs from module's class_name: issue `{module_name, issue: "class_mismatch", detail: "..."}` — **this check is required, not optional** (HW-22)
7. Return `{ok: bool, issues: [...]}`

Response shape:
```json
{
  "ok": false,
  "issues": [
    {
      "module_name": "Left_LED",
      "issue": "missing",
      "detail": "Pilot T has no config for hardware module Left_LED"
    }
  ]
}
```

### Step 3: Resolved hardware config endpoint

**File:** `api/routers/toolkits.py`

```
GET /api/toolkits/{id}/resolved-hardware-config/{pilot_id}
```

For each `hardware_module_id` in toolkit:
- Join → `hardware_modules` → get `name`, `class_name`
- Join → `hardware_libs` → get `filename` (as `lib_filename`)
- Join → `pilot_hardware_config` for `(pilot_id, module_id)` → get `config` dict
- Merge: `{class_name, lib_filename} | config`
- Key by module `name`

Response:
```json
{
  "hardware": {
    "Left_LED": {"class_name": "Digital_Out", "lib_filename": "gpio.py", "pin": 7, "polarity": 1},
    "Solenoid": {"class_name": "Solenoid", "lib_filename": "gpio.py", "pin": 12, "polarity": 1}
  }
}
```

### Step 4: Orchestrator — send resolved hardware config on START

**File:** `orchestrator/orchestrator/orchestrator_station.py`

`LOAD_HARDWARE_CONFIG` is an **outgoing** ZMQ message (orchestrator → Pi). Do NOT add an inbound handler in `orchestrator/main.py`.

Add `_send_hardware_config_if_needed()` and call it inside `start_run()` **after** `_send_hardware_libs_if_needed()` and **before** sending `START`. Do NOT call it inside dict-builder methods like `_build_step_task()` or `_build_first_step_task()` — those build the task payload but don't control message ordering.

```python
def _send_hardware_config_if_needed(self, pilot_name: str, pilot_id: int, toolkit_id: int | None):
    if toolkit_id is None:
        return
    hw_config = self.mics_api.get_resolved_hardware_config(pilot_id, toolkit_id)
    if not hw_config:
        return
    self.gateway.send(pilot_name, "LOAD_HARDWARE_CONFIG", {"hardware": hw_config})
```

Call sequence in `start_run()`:
```python
task_def = self.mics_api.get_task_definition(task_def_id)
toolkit_id = task_def.get("toolkit_id")
self._send_hardware_libs_if_needed(pilot_name, toolkit_id)
self._send_hardware_config_if_needed(pilot_name, pilot_id, toolkit_id)
self.gateway.send(pilot_name, "START", task_payload)
```

**API client method:**
```python
def get_resolved_hardware_config(self, pilot_id: int, toolkit_id: int) -> dict:
    resp = self._get(f"/api/toolkits/{toolkit_id}/resolved-hardware-config/{pilot_id}")
    return resp.get("hardware", {})
```

**File:** `~/pi-mirror/autopilot/autopilot/tasks/pilot.py`

Add `LOAD_HARDWARE_CONFIG` dispatch in the `elif msg_type ==` chain in `pilot.py` (NOT in `mics_task.py` — dispatch lives in pilot.py):


**File:** `~/pi-mirror/autopilot/autopilot/tasks/mics_task.py`

Initialize `_received_hw_config` in `__init__` so the attribute always exists:

```python
def __init__(self, *args, **kwargs):
    super().__init__(*args, **kwargs)
    self._received_hw_config: dict | None = None
    # ... rest of init
```

Extend `init_hardware()`:

```python
def init_hardware(self):
    if self._received_hw_config:
        self._init_hardware_from_config(self._received_hw_config)
    else:
        # Backward compat: use class-level HARDWARE dict (unchanged behavior)
        super().init_hardware()

def _init_hardware_from_config(self, config: dict):
    """Dynamically import + instantiate hardware classes from received config."""
    self.hardware = {}
    for name, spec in config.items():
        lib_module = spec["lib_filename"].replace(".py", "")
        class_name = spec["class_name"]
        # Guard: override dir must be in sys.path (added by receive_hardware_libs)
        override_dir = str(Path.home() / "apps" / "hardware_overrides")
        if override_dir not in sys.path:
            raise RuntimeError(
                f"Hardware override dir not in sys.path — LOAD_HARDWARE_LIBS must be sent before LOAD_HARDWARE_CONFIG"
            )
        mod = importlib.import_module(lib_module)
        cls = getattr(mod, class_name)
        kwargs = {k: v for k, v in spec.items() if k not in ("class_name", "lib_filename")}
        self.hardware[name] = cls(**kwargs)
```

Backward compat guaranteed: if no `LOAD_HARDWARE_CONFIG` received, `_received_hw_config` is None and `super().init_hardware()` is called (uses `self.HARDWARE` class constant — unchanged behavior).

### Step 6: Orchestrator — stable promotion on INC_TRIAL_COUNTER (deferred from Phase 09, step 4d)

This step runs after the end-to-end flow is verified (Steps 1–5), closing the hardware lib lifecycle: a real trial run on hardware is the final confirmation that a lib version is production-stable.

**File:** `orchestrator/orchestrator/orchestrator_station.py`

In the existing `_handle_inc_trial()`, after the graduation check, add stable promotion:

```python
def _handle_inc_trial(self, value: dict):
    subject_key = value.get("subject")
    if not subject_key:
        return
    run = self.api.get_run_by_subject_key(subject_key)
    if not run or run.get("status") != "running":
        return
    resp = self.api.increment_trial(run["id"])
    if resp.get("should_graduate"):
        self._advance_run_step(run)

    # Stable promotion: a real trial run confirms the hw lib works on hardware
    pilot = self.api.get_pilot(run["pilot_id"])
    if pilot:
        pilot_key = self.state.resolve_pilot_key(
            db_name=pilot.get("name"), ip=pilot.get("ip")
        )
        active_run = self.state.get_active_run(pilot_key)
        if active_run and active_run.get("toolkit_id"):
            self.api.promote_active_hw_libs_to_stable(active_run["toolkit_id"], pilot_key)
```

Note: `pilot_name`/`pilot_key` must be resolved via `run["pilot_id"]` — only `subject_key` is in scope from the ZMQ message. Same pattern as `_advance_run_step()`.

**File:** `orchestrator/orchestrator/mics/mics_api_client.py`

```python
def promote_active_hw_libs_to_stable(self, toolkit_id: int, pilot_name: str):
    libs = self.get_toolkit_hardware_libs(toolkit_id)
    for lib in libs:
        if lib.get("state") == "beta":
            self._patch(
                f"/api/hardware-libs/{lib['id']}/mark-stable",
                {"reason": "protocol_run", "pilot": pilot_name}
            )
```

Only `beta` libs are promoted — already-`stable` versions are a no-op. `unvalidated` versions are skipped (they should not have been deployed).
