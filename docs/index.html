User Guide

System Overview

Traditional experimental approaches require frequent manual intervention, introduce experimenter-dependent variability, and limit throughput, duration, and reproducibility. MICS was built to remove these constraints.

MICS (Modular Interactive Control System) is an open-source platform for automated, scalable experimental control. It separates interactive task logic from hardware implementation: sensors, effectors, and external systems are abstracted as interchangeable functional units, allowing complex experimental environments to be constructed and modified without low-level engineering effort.

Experimental flow is executed through finite state machine logic — called Blueprints — running on dedicated Raspberry Pi controllers ("pilots"). This provides deterministic control at millisecond resolution while supporting adaptive task progression driven by subject behavior. A central orchestration layer coordinates experiment definition, live monitoring, and cohort-level data integration across all setups simultaneously.

MICS is designed for seamless integration with neuroscience technologies including optogenetics, electrophysiology, calcium imaging, and video-based behavioral tracking. Event-locked triggers and closed-loop interactions can be embedded directly in task logic, allowing behavioral contingencies to interact dynamically with external measurement or stimulation systems. The same infrastructure supports purely behavioral studies, chronic home-cage monitoring, circadian-scale experiments, and large-cohort automated training.

The Four Services

Web UI :8080
The browser interface you interact with. Hosts the React SPA at /react/ and proxies requests to the API and Orchestrator on your behalf.
Orchestrator :9000
The gateway between the web and the Pi devices. Receives ZMQ messages from pilots, maintains live state in Redis, streams events to Elasticsearch, and calls the API for trial increments and graduation checks.
API :8000
The REST backend. Stores subjects, protocols, sessions, and run progress in PostgreSQL. JWT-authenticated; the Web UI holds the token — you never need it directly.
Backup —
Runs a nightly job at 23:59 that takes a PostgreSQL dump and an Elasticsearch snapshot, then uploads them to the lab SMB share on storwiz.

Pi Setup & Onboarding

Each Raspberry Pi arrives as a pre-configured OS image (TAR archive) stored on the lab SMB server. Burning it produces a Pi that is identical to all others — network config, autopilot software, and task plugins are already in place.

⚠
Image source The Pi OS image is stored on storwiz at:
\\isi.storwis.weizmann.ac.il\labs\yizharlab\Mics\
Connect to the Weizmann VPN (or be on campus) before accessing it.

Step-by-step: Burning a New Pi

  1. 1
    Get the image

    Mount the SMB share and copy the .tar Pi image file to your local machine. The file is a full SD card image compressed as a TAR archive.

  2. 2
    Extract the image

    Unpack the TAR to get the raw .img file. On Windows, use 7-Zip or WinRAR. On macOS/Linux:

    bash
    tar -xf mics_pi_image.tar
  3. 3
    Open Raspberry Pi Imager

    Download and install Raspberry Pi Imager if you don't have it. Launch it and insert your SD card (≥16 GB recommended).

  4. 4
    Select the custom image

    Click Choose OS → scroll to the bottom → Use custom → select the .img file you extracted in step 2.

  5. 5
    Select the SD card

    Click Choose Storage and select your SD card. Double-check it is the correct device before proceeding.

    ⛔
    Select the right device. Writing to the wrong drive will destroy its data with no undo.
  6. 6
    Write

    Click Write and confirm. Raspberry Pi Imager will flash and verify the image. This takes a few minutes depending on SD card speed.

  7. 7
    Boot and connect

    Insert the SD card into the Pi, connect ethernet, and power on. The autopilot service starts automatically on boot. Within ~30 seconds the Pi sends a HANDSHAKE to the orchestrator and appears in the pilot grid.

  8. 8
    Verify in the UI

    Open http://localhost:8080/react/. The new pilot should appear as connected within a minute. Its available tasks are auto-discovered from the HANDSHAKE message.

ℹ
Identical setup, every time Because every Pi is burned from the same TAR image, all configuration (network settings, task plugins, autopilot version, OS config) is guaranteed identical. No manual setup is required after burning.

Managing Subjects

A subject is an animal enrolled in the system. Each subject has a unique name and tracks which protocol it should run next.

Creating a Subject

  1. 1
    Navigate to Subjects

    Go to http://localhost:8080/react/subjects-ui or click Subjects in the navigation bar.

  2. 2
    Enter a name

    Type the subject name in the Create Subject form. Convention is m_001, m_002, … but any unique string is valid.

  3. 3
    Submit

    Click Create. The subject appears in the list immediately and is ready to be assigned a protocol.

Assigning a Protocol

Navigate to /react/protocols-ui, find the protocol you want, and use the assignment panel on the right to select a subject and assign. This creates a session for that subject with the chosen protocol as its blueprint.

ℹ
One active protocol at a time Each subject has a next_protocol pointer. Assigning a new protocol while one is already active updates that pointer — the new protocol takes effect on the next session start.

Managing Protocols

A protocol (also called a blueprint) is a sequence of task steps. Each step specifies a task type, display name, parameters, and a graduation rule that determines when the subject advances to the next step.

Creating a Protocol

  1. 1
    Open the Protocol Builder

    Navigate to /react/protocols-create. You'll see a two-panel view: available task types on the left, protocol builder on the right.

  2. 2
    Add steps from the palette

    Click a task type in the left panel to add it as a step. Set the step name and any parameter overrides. Steps run in order from top to bottom.

  3. 3
    Configure graduation

    Each step has a graduation rule embedded in its parameters. The most common type is n_trials — the subject moves to the next step after reaching a trial count threshold.

  4. 4
    Name and save

    Give the protocol a name and click Save Protocol. It is immediately available for assignment to subjects.

Step Parameters

Parameters control task behavior (timing, thresholds, reward amounts). Each task type exposes its own parameter schema — the UI shows the available fields automatically when a task type is selected. The graduation, step_name, and task_type keys are reserved and shown separately.

Running Sessions

A session pairs a subject with a protocol. A session run is a single execution attempt on a specific pilot. You can start, stop, resume, and restart runs from the session card view.

Starting a Run

  1. 1
    Find the session

    Navigate to the pilot's session list at /react/pilots/:pilotName/sessions-ui, or to a subject's session list at /react/subjects/:subjectName/sessions-ui.

  2. 2
    Choose a pilot and mode

    On the subject's page, select which pilot to run on and choose a start mode. On the pilot's page, click START on the session card.

  3. 3
    Optionally override parameters

    Click the session title (protocol name) to open the Overrides Modal. Set per-step parameter overrides before clicking START — these override the protocol defaults for this run only.

  4. 4
    Confirm and start

    Click START. The orchestrator sends the task payload to the Pi. The pilot status changes to running within seconds.

Start Modes

ModeWhat it doesWhen to use
new Creates a fresh run starting from step 0, trial 0 Normal start; subject has never run this session
resume Reactivates the most recent stopped or errored run in place — same step, same trial count Session was interrupted (power cut, task error); continue from exact point
restart Creates a new run row but copies the current step index — restarts the step from trial 0 Trial data was bad and you want to redo the current step cleanly

Stopping a Run

Click the STOP button on the pilot card (pilot grid page) or on the session card. The orchestrator sends a stop command to the Pi. The run status changes to stopped. The subject's progress is preserved and can be resumed.

  1. 1
    Navigate to Researchers

    Click Researchers in the navigation bar or go to /react/researchers-ui.

  3. 3

Editing a Researcher

Click the Edit button on any researcher row. An inline form appears where you can update the name and/or email. Click Save to confirm. If you enter a duplicate email the API returns a clear error message (e.g. "A researcher with this email already exists").

Removing a Researcher

Open the inline Edit form and click Remove. This performs a soft-delete — the record remains in the database with is_hidden = true and disappears from the normal list. Hidden records can be retrieved via the API by passing ?include_hidden=true. This preserves referential integrity for subjects that were linked to the researcher.

ℹ
Soft-delete is permanent from the UI There is currently no "undelete" button in the UI. To restore a hidden researcher, use the API directly or a database query to set is_hidden = false.

Architecture

MICS is a set of five Docker containers orchestrated by Compose, plus supporting infrastructure (PostgreSQL, Redis, Elasticsearch) and one or more Raspberry Pi devices on the local network.

The Orchestrator uses network_mode: host so it can reach both the local Docker network (API) and the Pi devices on the physical network via ZMQ without NAT complications.

Elasticsearch runs outside Docker (directly on the host) and is accessed from containers via host.docker.internal:9200.

Technical Reference

Services

API (api/)

FastAPI application served by Uvicorn on port 8000. Owns the persistent data model. All routes require a JWT Bearer token (HS256, audience mics-clients, issuer mics-api). The Web UI holds the token; browser clients never see it.

Uses a dual ORM: SQLModel manages newer tables (subjects, protocols); SQLAlchemy manages legacy tables (pilots, sessions, session runs, run progress, task definitions). Both share the same PostgreSQL engine.

Orchestrator (orchestrator/)

FastAPI + Tornado + PyZMQ. Runs a ZMQ ROUTER socket on port 5560 that Pi pilots connect to as DEALER sockets. Incoming messages are dispatched to handler methods by message key. Write-heavy operations (Elasticsearch ingestion) are handled by worker threads draining bounded queues (maxsize=50,000) to prevent IOLoop stalls.

Live pilot state is maintained as Redis hashes (pilot:{name}). The Web UI polls GET /pilots/live every second and pushes updates to connected browsers over WebSocket.

Web UI (web_ui/)

FastAPI reverse proxy on port 8080. Authenticates all upstream calls to the API and Orchestrator using the MICS_API_TOKEN env var. Browser clients never touch API credentials directly.

Also serves the React SPA at /react/ — a Vite-built TypeScript application using React 18, React Router, and TanStack React Query. Pilot state is delivered to the browser via a WebSocket at /ws/pilots.

Backup (backup/)

Supercronic-scheduled container. Runs at 23:59 (Asia/Jerusalem) every night:

  1. Runs pg_dump against the PostgreSQL service and compresses it
  2. Triggers an Elasticsearch snapshot to the configured repository
  3. Uploads both to the SMB share at \\isi.storwis.weizmann.ac.il\labs\yizharlab\Mics\database_backup
Credentials are injected via Docker secrets (never in environment variables).

Technical Reference

ZMQ Protocol

Pi pilots connect to the orchestrator's ZMQ ROUTER socket on port 5560 using DEALER sockets. Each pilot's DEALER identity is its configured NAME. The orchestrator's own name is "T".

All messages are multi-part ZMQ frames. The orchestrator dispatches on the message key (second frame), routing to the corresponding handler method.

Message Types

HANDSHAKE
Pi → Orchestrator. Sent on pilot startup. Contains the pilot's name, available task classes (with PARAMS, HARDWARE, FLAGS), and hardware configuration. The orchestrator upserts the pilot in PostgreSQL and populates task_definitions.
PING
Pi → Orchestrator. Keepalive. Updates the pilot's updated_at timestamp in Redis. Used for the 15-second staleness check.
STATE
Pi → Orchestrator. Pilot state transition (e.g. IDLE → RUNNING → IDLE). Updates the Redis state field and propagates to the browser via WebSocket.
CONTINUOUS / STREAM
Pi → Orchestrator. High-frequency behavioral events — hardware sensor reads, lick detection, timer firings, state transitions. Both keys route to on_data(). Events are enqueued and written to Elasticsearch by worker threads.
INC_TRIAL_COUNTER
Pi → Orchestrator. Sent explicitly by the task when a trial completes. Triggers the graduation check: the orchestrator calls the API to increment the trial count and checks if the subject should advance to the next protocol step.
DONE
Pi → Orchestrator. Task has finished all its work cleanly. Marks the session run as completed.
⚠
INC_TRIAL_COUNTER is never automatic Every Pi task must send this message explicitly at the end of each trial. If a subject is not graduating, verify that the task code dispatches this message and that it is reaching the orchestrator (check orchestrator logs).
Technical Reference

Pi Task System

Each behavioral task runs on the Pi as a Python class. Tasks are discovered automatically via HANDSHAKE — no manual registration required. All task files live in ~/Apps/mice_interactive_home_cage/terminal/plugins/ on the Pi.

Class Hierarchy

python
Task  (autopilot base)
  └── mics_task  (ZMQ comms, HANDSHAKE auto-registration)
        └── learning_cage  (standard cage hardware defaults)
              └── YourConcreteTask  (in terminal/plugins/)

Task Declaration

Every task class declares three class-level dicts that are read at HANDSHAKE time and stored in the DB:

python
class AppetitiveTask(learning_cage):

    # Tunable parameters — shown in the UI param editor
    PARAMS = {
        'open_duration':  {'tag': 'Reward open (ms)',    'type': float, 'default': 50},
        'iti_duration':   {'tag': 'ITI duration (ms)',    'type': float, 'default': 5000},
        'lick_threshold': {'tag': 'Lick count to reward', 'type': int,   'default': 3},
    }

    # Physical hardware — GPIO pins, I2C devices
    HARDWARE = {
        'REWARD': {'type': 'Solenoid',   'pin': 11},
        'LICKER': {'type': 'LickSensor', 'pin': 7},
        'IR2':    {'type': 'Beambreak',  'pin': 13},
    }

    # Tracker flags — binary/counted state (lick count, hit/miss, etc.)
    FLAGS = {
        'hit_trial':    BooleanTracker,
        'lick_count':   CountTracker,
    }

State Machine (FDA)

Task flow is a FiniteDeterministicAutomaton. Each stage method ends with return self.wait_for_condition(), which blocks until a registered transition fires. Transitions are declared as lists of lambda conditions over self.view:

python
def __init__(self, **kwargs):
    super().__init__(**kwargs)

    self.stages = FiniteDeterministicAutomaton()
    self.stages.add_method(self.wait_for_lick)
    self.stages.add_method(self.deliver_reward)
    self.stages.add_method(self.iti)

    # Transition: wait_for_lick → deliver_reward when LICKER fires
    self.stages.add_transition(
        self.wait_for_lick, self.deliver_reward,
        [lambda: self.view.get_value('LICKER') == True]
    )

def wait_for_lick(self):
    self.hardware['REWARD'].trigger()   # auto-logged via @log_action
    return self.wait_for_condition()     # blocks here

def deliver_reward(self):
    # Graduation check — must be sent explicitly
    self.node.send('T', 'INC_TRIAL_COUNTER', {'n_trials': self.trial_counter})
    return self.wait_for_condition()

Auto-logging

All methods on Hardware and Tracker classes are decorated with @log_action. This automatically dispatches a CONTINUOUS ZMQ event whenever hardware is activated — no manual logging code required in task methods.

Technical Reference

Session Run Modes

The mode field on a session_run controls how progress is initialized when a task starts:

ModeDB behaviorPi payload
new New session_run row + new run_progress at step 0, trial 0 Fresh start; no prior state sent to Pi
resume Reactivates the most recent STOPPED or ERROR run — no new rows created Sends current_step_idx and current_trial from existing run_progress
restart New session_run row with same session_run_index as the previous run (records the retry). New run_progress at same step, trial 0 Same step, fresh trial counter

Override Layering

Overrides set in the UI are stored on session_run.overrides as a JSON object with this shape:

json
{
  "global": {
    "iti_duration": 3000   // applies to all steps
  },
  "steps": {
    "0": { "lick_threshold": 1 },  // step-specific wins over global
    "2": { "lick_threshold": 5 }
  }
}

The orchestrator applies global overrides first, then step-specific values (which win). The merged result is sent to the Pi in the START payload.

Schedule

Backups run at 23:59 Israel time every night (BACKUP_SCHEDULE="59 23 * * *", timezone Asia/Jerusalem).

Restoring PostgreSQL

bash
# Copy the dump from SMB to local, then restore:
gunzip mics_db_YYYY-MM-DD.sql.gz
docker exec -i mics_db psql -U mics_user -d mics_db < mics_db_YYYY-MM-DD.sql

Restoring Elasticsearch

bash
# Register the repository (if not already registered):
curl -X PUT http://localhost:9200/_snapshot/org_repo \
  -H 'Content-Type: application/json' \
  -d '{"type":"fs","settings":{"location":"/backups"}}'

# Restore a specific snapshot:
curl -X POST http://localhost:9200/_snapshot/org_repo/<snapshot_name>/_restore
ℹ
Elasticsearch runs outside Docker ES is not containerized. Its snapshot directory is bind-mounted from the host at /home/ido/Elastic_snapshots. Ensure this path exists before starting the stack.
