# Phase 11, Plan 1: Toolkit Redesign (Backend-Authored)

**Requirements:** HW-12, HW-13, HW-14, HW-15, HW-16
**Phase goal:** Toolkits are fully backend-defined. HANDSHAKE populates available_locked_states. User assembles a toolkit from locked states + hardware modules + flags + params via a 5-step UI.

**Gate:** Requires Phase 10 complete — `hardware_modules` table must exist before toolkits can reference `hardware_module_ids`.

---

## Context

Currently toolkits are created entirely by HANDSHAKE — the Pi announces its task class and the orchestrator stores the metadata. This is read-only from the backend perspective. Phase 11 flips this: researchers author toolkits in the backend UI, picking from:
1. **Locked states** — state method names from a Pi task file (announced via HANDSHAKE)
2. **Hardware modules** — global module records from Phase 10
3. **Flags** — named boolean/numeric flags
4. **Params** — named typed parameters with defaults

HANDSHAKE-registered toolkits remain valid (`is_backend_authored=FALSE`) and show a "legacy" badge.

def _put(self, path: str, body: dict) -> dict:
    resp = self.session.put(f"{self.base_url}{path}", json=body, headers=self._auth_headers())
    resp.raise_for_status()

### Step 5: HANDSHAKE update — populate available_locked_states

**File:** `orchestrator/orchestrator/orchestrator_station.py`

Updated HANDSHAKE handler accepts both formats:

**New format** (Pi with updated code):
```json
{
  "tasks": [
    {"filename": "appetitive.py", "state_names": ["init_trial", "reward_state", "timeout_state"]}
  ]
}
```

**Legacy format** (old Pis — still accepted):
```json
{
  "STAGE_NAMES": ["init_trial", "reward_state", "timeout_state"],
  "task_type": "AppetitiveTaskReal"
}
```

**Legacy filename caveat:** `f"{task_type}.py"` produces `AppetitiveTaskReal.py` — this is the Python class name, not the Pi source filename (`appetitive.py`). Store the reconstructed value in `task_filename` as-is, but mark it clearly in the available_locked_states row (e.g., via a separate `is_legacy_filename: bool` column or by convention). The toolkit creation UI must display a warning when a task source file comes from a legacy HANDSHAKE, since the filename may not match what a researcher expects. Do NOT silently conflate class names with filenames.

For new format: for each `{filename, state_names}`, upsert via:
```python
self.mics_api.upsert_locked_states(pilot_id, filename, state_names)
```

For legacy format: reconstruct filename as `f"{task_type}.py"` and upsert similarly.

**API client method (uses `_put()` from Step 1):**
```python
def upsert_locked_states(self, pilot_id: int, task_filename: str, state_names: list[str]):
    self._put(f"/api/locked-states/{pilot_id}/{task_filename}", {"state_names": state_names})
```

**Modal content:**

1. **Header:** `{pin.lib_filename}` — current status chip (pinned or active)

2. **Version picker:** Load `GET /api/hardware-libs/{pin.hardware_lib_id}/versions` → dropdown sorted by `version_number DESC`
   - Each option: `v{version_number} — {state} — {created_at date}`
   - Default selection: current pinned version, or active version if not pinned

3. **On version select** (not confirm — on dropdown change):
   - Determine `currentVersionId`: `pin.pinned_version_id ?? pin.active_version_id`
   - If `selectedVersionId !== currentVersionId` AND both are non-null:
     - Fetch `GET /api/hardware-libs/{lib_id}/versions/diff?from={currentVersionId}&to={selectedVersionId}`
     - Run client-side FDA scan (see below)
     - Display result
   - If `selectedVersionId === currentVersionId`: clear any warnings

4. **FDA scan** (client-side, pure function):
   ```ts
   function scanFdaForLibRefs(fdaJson: FdaJson, toolkit: ToolkitRead, libFilename: string): ScanRef[] {
     // Walk all states.entry_actions recursively (including type:if branches)
     // Match type:"hardware" and type:"timer" actions where the module's lib filename === libFilename
     // toolkit.hardware_module_ids → need module details (class → lib) to resolve filename
     // Return: [{state_name, action_type, ref (module name), method}]
   }
   ```
   Note: to map module name → lib filename, we need the hardware modules + their lib. This data is already available via `toolkit.hardware_module_ids` — fetch `GET /api/hardware-modules/{id}` for each (or cache from a parent query).

5. **Warning panel** (shown between dropdown and confirm button):
   ```tsx
   {diff && (diff.removed_methods.length > 0 || diff.changed_signatures.length > 0) && (
     <div className="badge status-error" style={{display: 'block', marginTop: 8}}>
       <strong>⚠ Potential breakage in this FDA:</strong>
       <ul>
         {affectedRefs.map(r => (
           <li key={`${r.state_name}-${r.method}`}>
             State "{r.state_name}": {r.ref}.{r.method}
             {removedSet.has(r.method) ? " removed" : " signature changed"}
           </li>
         ))}
       </ul>
       {affectedRefs.length === 0 && diff.removed_methods.length > 0 && (
         <span>Removed methods not referenced in this FDA — safe to upgrade.</span>
       )}
     </div>
   )}
   ```

6. **Action buttons:**
   - **Set Pin** (`button-primary`) — `PUT /api/task-definitions/{taskDefId}/hw-lib-pins/{lib_id}` with selected version → `onSaved()`
   - **Revert to active** (`button-secondary`, shown only if currently pinned) — `DELETE` pin → `onSaved()`
   - **Cancel** (`button-secondary`) → `onClose()`
