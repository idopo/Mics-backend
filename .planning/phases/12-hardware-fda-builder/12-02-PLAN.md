# Phase 12, Plan 2: Impact Detection + Broken Definition Backend

**Requirements:** HW-17, HW-18, HW-19, HW-20
**Phase goal:** Backend detects when hardware lib or toolkit changes break existing task definitions and flags them. Task definition save re-validates.

**Scope:** Backend only — DB columns, migration, impact scan logic, and save re-validation. UI display is in Plan 12-01.

**Gate:** Requires Plan 12-01 types to be in place (validation_status field expected by UI).

### Step 2: `scan_fda_for_refs()` helper

**File:** `api/routers/hardware_libs.py` (or a shared `api/fda_utils.py`)

Recursive walker that returns a flat list of all action refs in an FDA JSON:

```python
def scan_fda_for_refs(fda_json: dict) -> list[dict]:
    """Recursively walk all states.entry_actions, including type:if branches.

    Returns list of {state_name, action_type, ref, method} for hardware/flag/timer/method actions.
    """
    results = []
    for state_name, state_body in fda_json.get("states", {}).items():
        results.extend(_scan_actions(state_name, state_body.get("entry_actions", [])))
    return results

def _scan_actions(state_name: str, actions: list[dict]) -> list[dict]:
    results = []
    for action in actions:
        action_type = action.get("type")
        if action_type in ("hardware", "flag", "timer", "method"):
            results.append({
                "state_name": state_name,
                "action_type": action_type,
                "ref": action.get("ref") or action.get("action"),
                "method": action.get("method"),
            })
        elif action_type == "if":
            results.extend(_scan_actions(state_name, action.get("then", [])))
            results.extend(_scan_actions(state_name, action.get("else", [])))
    return results
```

### Step 3: Impact detection — HardwareLib updated

**File:** `api/routers/hardware_libs.py` — extend `PUT /api/hardware-libs/{id}`

On update:
1. Store old `ast_metadata` before overwrite (use version's `ast_metadata` if set, else re-parse `source_code`)
2. Re-validate + re-extract new AST metadata
3. Diff per class: find methods present in old AST but absent in new (removed or renamed)
4. If any methods removed:
   a. Query all `toolkit_hardware_libs` rows referencing this lib → get toolkit_ids
   b. Query all task_definitions linked to those toolkits
   c. For each task definition:
      - **Pin check:** query `task_definition_hw_lib_pins` where `(task_def_id, hardware_lib_id)` matches
      - **If pinned to a specific version:** skip this task definition entirely — it is insulated from active-version changes. Optionally log: `"task_def {id} is pinned to v{version_number}; active lib changed — no impact flagged"`
      - **If NOT pinned (uses active):** call `scan_fda_for_refs(fda_json)`, check if any `{type: "hardware", ref, method}` entry uses a removed method
      - If match found: set `validation_status = "broken"`, `validation_message = f"State '{state_name}': {ref}.{method} removed from lib"`
5. Return diff + list of affected (and skipped-pinned) definition IDs in response

```python
removed_methods: dict[str, set[str]] = {}   # class_name → {method_names}
for cls in old_classes:
    old_set = {m["name"] for m in cls["methods"]}
    new_cls = next((c for c in new_classes if c["name"] == cls["name"]), None)
    new_set = {m["name"] for m in new_cls["methods"]} if new_cls else set()
    removed = old_set - new_set
    if removed:
        removed_methods[cls["name"]] = removed

# Pin-aware impact scan
pinned_task_def_ids = {
    row.task_def_id
    for row in db.query(TaskDefinitionHwLibPin)
    .filter(TaskDefinitionHwLibPin.hardware_lib_id == lib_id)
    .all()
}
for task_def in affected_task_defs:
    if task_def.id in pinned_task_def_ids:
        continue  # insulated — pin shields from active version change
    refs = scan_fda_for_refs(task_def.fda_json or {})
    # ... flag broken if removed method found in refs
```

**Important:** This means pinned task definitions accumulate silent drift — their FDA may reference methods removed in newer lib versions, but they won't be flagged because they're locked to an older version. The Phase 11 `HwLibVersionModal` exposes this when the researcher tries to upgrade: the AST diff warnings will show which states will break. This is the intended design (researcher controls upgrade timing).

### Step 4: Impact detection — Toolkit changed

**File:** `api/routers/toolkits.py` — extend `PUT /api/toolkits/{id}`

When a toolkit's `flags` or `hardware_module_ids` changes:

**Removed flag:** scan linked task definitions for:
- `{type: "flag", ref: <flag_name>}` in any action
- `{flag: <flag_name>}` in any condition operand or arg

**Removed hardware module:** scan for:
- `{type: "hardware", ref: <module_name>}`
- `{type: "timer", ref: <module_name>}`

For each affected task definition: set `validation_status = "broken"`, `validation_message` with specific state + ref.

### Step 5: Task definition save re-validates

**File:** `api/routers/task_definitions.py` (or `api/main.py`) — extend `PUT /api/task-definitions/{id}`

On save:
1. Call `scan_fda_for_refs(new_fda_json)` to get all refs
2. For each hardware ref: verify `{ref, method}` exists in current AST of the linked lib
3. For each flag ref: verify flag name exists in the toolkit's current flags
4. If no broken refs found: set `validation_status = "ok"`, `validation_message = null`
5. If broken refs found: update `validation_status = "broken"`, `validation_message` accordingly

This means saving a corrected task definition automatically clears the broken status.
