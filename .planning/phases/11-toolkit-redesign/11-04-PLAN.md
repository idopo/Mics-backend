# Phase 11, Plan 4: FLAGS + PARAMS Backend Injection

**Gap closure:** Phase 11-03 planned and claimed FLAGS/PARAMS injection but did not implement it.
The orchestrator has an explicit suppression comment; `mics_task.__init__` only handles HARDWARE
and PREFS_HARDWARE. This plan implements what 11-03 left incomplete.

**Phase goal:** Backend sends FLAGS and PARAMS in the START payload. Pi resolves tracker type
strings to real Tracker classes and applies them before `init_flags()` runs. After this plan, a
backend-authored toolkit needs no Pi-side FLAGS or PARAMS declarations.

### Step 3: Pi — add `_resolve_flags()` static method

**File:** `~/pi-mirror/autopilot/autopilot/tasks/mics_task.py`
**Location:** After `_coerce_hw_value()`

```python
@staticmethod
def _resolve_flags(flags_dict: dict) -> dict:
    """Map string tracker type names to Tracker classes for backend-injected FLAGS."""
    TYPE_MAP = {
        "Counter_Tracker": Tracker.Counter_Tracker,
        "Boolean_Tracker": Tracker.Boolean_Tracker,
        "Trial_Tracker":   Tracker.Trial_Tracker,
    }
    resolved = {}
    for name, spec in flags_dict.items():
        spec = dict(spec)
        type_str = spec.get("type", "Counter_Tracker")
        if type_str not in TYPE_MAP:
            logging.getLogger(__name__).warning(
                "Unknown tracker type %r for flag %r — falling back to Counter_Tracker", type_str, name
            )
        spec["type"] = TYPE_MAP.get(type_str, Tracker.Counter_Tracker)
        resolved[name] = spec
    return resolved
```

`Tracker` is already imported at module level (`from autopilot.utils.Tracker import Tracker`).
Use the module-level `logging` import rather than a local one.
