## Context

After Phase 9 we have hardware libs stored with AST metadata. The Phase 09 AST extractor walks all `FunctionDef` nodes including `__init__`, so constructor signatures are captured and available for Phase 10's config editor.

Now we introduce the semantic layer:

- **HardwareModule** = a global named device (e.g., `Left_LED ã gpio.Digital_Out`). Not pilot-specific. Referenced by toolkits.
- **PilotHardwareConfig** = per-pilot binding mapping a HardwareModule to physical config (`{pin, polarity, duration, ...}`). Replaces the `HARDWARE` section of prefs.json.

The HANDSHAKE handler seeds pilot hardware config from prefs.json on first boot.

### Step 3: Hardware Modules API

**File:** `api/routers/hardware_modules.py` (new)

```
GET    /api/hardware-modules              ã list all (id, name, display_name, hardware_lib_id, class_name)
POST   /api/hardware-modules              ã create; validate class_name exists in lib's AST metadata
GET    /api/hardware-modules/{id}         ã full detail
PUT    /api/hardware-modules/{id}         ã update; re-validate class_name
DELETE /api/hardware-modules/{id}         ã delete (blocked if referenced by toolkit hardware_module_ids)
GET    /api/hardware-modules/{id}/methods ã return method list from linked lib's AST for class_name
```

On create/update: query `hardware_libs.ast_metadata["classes"]` where `name == class_name` ã if not found, return 422 "class_name not found in lib AST".

`GET /api/hardware-modules/{id}/methods` returns the full method list for the module's class, pulled from the linked lib's `ast_metadata`. Includes `__init__` so the config editor (Plan 10-02) can derive constructor arg fields:

```json
{
  "module_id": 1,
  "module_name": "Left_LED",
  "class_name": "Digital_Out",
  "methods": [
    {"name": "__init__", "args": [
      {"name": "pin", "annotation": "int"},
      {"name": "polarity", "annotation": "int", "default": "1"}
    ]},
    {"name": "on", "args": []},
    {"name": "off", "args": []},
    {"name": "pulse", "args": [{"name": "duration", "annotation": "float", "default": "0.05"}]}
  ]
}
```

### Step 4: Pilot Hardware Config API

**File:** `api/routers/pilot_hardware_config.py` (new)

```
GET    /api/pilots/{pilot_id}/hardware-config                    ã list all config rows for pilot
PUT    /api/pilots/{pilot_id}/hardware-config/{module_id}        ã upsert config for module
DELETE /api/pilots/{pilot_id}/hardware-config/{module_id}        ã remove module config
POST   /api/pilots/{pilot_id}/hardware-config/seed               ã seed from raw HARDWARE dict (orchestrator-internal)
```

`PUT` body: `{config: {pin: 7, polarity: 1, ...}}`

`POST /seed` body (called by orchestrator, not user-facing):
```json
{
  "hardware": {
    "Left_LED": {"class": "gpio.Digital_Out", "pin": 7, "polarity": 1},
    "Solenoid": {"class": "gpio.Solenoid", "pin": 12, "polarity": 1}
  }
}
```

Seeder logic: for each key in `hardware`, look up HardwareModule by name ã if found and pilot has no config row for it, INSERT the config. If module not found by name, log warning and skip (don't auto-create modules). If config rows already exist, skip entirely (one-time migration).

### Step 5: HANDSHAKE update ã seed pilot hardware config

**File:** `orchestrator/orchestrator/orchestrator_station.py`

In HANDSHAKE handler: HARDWARE lives inside `payload["prefs"]` (the Pi sends `{'pilot': ..., 'prefs': prefs.get(), 'tasks': ...}`). Add the seed call inside the existing `try` block, after `create_or_update_pilot()`.

**Critical:** Use `pilot_obj["id"]` to obtain `pilot_id` ã `pilot_obj` is the return value of `create_or_update_pilot()` already called earlier in `on_handshake()`. Do NOT call `self.state.get_pilot_id(pilot_name)` ã that method does not exist on `OrchestratorState`. Variable name in the handler is `payload` (not `handshake_data`):

```python
hardware_section = payload.get("prefs", {}).get("HARDWARE", {})
if hardware_section:
    self.mics_api.seed_pilot_hardware_config(pilot_obj["id"], hardware_section)
```

**Mics API client method:**
```python
def seed_pilot_hardware_config(self, pilot_id: int, hardware: dict):
    self._post(f"/api/pilots/{pilot_id}/hardware-config/seed", {"hardware": hardware})
```

```python
def handle_hardware_lib_test_result(self, msg: Message):
    data = msg.value or {}
    pilot_name = msg.sender
    version_id = data.get("version_id")
    ok = data.get("ok", False)
    error = data.get("error")
    self.api.patch_hardware_lib_version(
        version_id, ok=ok, error=error, pilot=pilot_name
    )
```

**File:** `orchestrator/orchestrator/mics/mics_api_client.py`

```python
def patch_hardware_lib_version(self, version_id: int, ok: bool, error: str | None, pilot: str):
    self._patch(
        f"/api/hardware-libs/versions/{version_id}/validate",
        {"ok": ok, "error": error, "pilot": pilot}
    )
```

This closes the test-import feedback loop: when the orchestrator sends `LOAD_HARDWARE_LIBS` with `test_import: True`, the Pi runs the import and replies with `HARDWARE_LIB_TEST_RESULT`; this handler writes the result to the DB (updating version state to `beta` or back to `unvalidated`).
