# Phase 12, Plan 1: Full FDA State Builder UI (Hardware + Flag + Param + Conditional)

**Requirements:** HW-17, HW-18, HW-19, HW-20
**Phase goal:** FDA state builder supports the complete entry_action vocabulary in the UI. Impact detection backend is in Plan 12-02.

**Scope:** This plan covers the React UI only — full action editor in StateBodyPanel + type updates + broken-definition display. Backend validation columns and impact detection logic are in Plan 12-02.

**Gate:** Requires Phase 10 (hardware modules + /methods endpoint) and Phase 11 (backend-authored toolkits with typed flags and hardware_module_ids) complete.

---

## Context

A state's `entry_actions` list is a **sequence of heterogeneous actions** that executes when the state is entered on the Pi. The Pi's `mics_task._build_action_callable()` already handles all six action types — the state builder must expose all of them so researchers can compose states visually without writing Python.

The complete vocabulary (from FDA JSON v2 schema and `mics_task.py`):

| type | ref points to | what it does on Pi |
|---|---|---|
| `hardware` | hardware module name | calls `self.hardware[group][id].method(*args)` |
| `flag` | flag name (from toolkit flags) | calls tracker method; available methods depend on tracker type |
| `timer` | timer hardware module name | calls timer.start/stop/reset with duration |
| `method` | locked state / callable method name | calls `self.method_name(*args)` on the task instance |
| `if` | — | evaluates condition → executes then-branch or else-branch actions |

`type: special` (previously used for `INC_TRIAL_COUNTER`) is **legacy only** — backward compat for existing FDA JSONs. The state builder does not expose it. Trial counting is done by calling `increment()` on a `Trial_Tracker` flag, which internally dispatches `key='INC_TRIAL_COUNTER'` to the orchestrator.

Arg values resolved at run-time:
- `{"literal": value}` — hardcoded value
- `{"param": "name"}` — resolved from `self.params[name]` at call time
- `{"flag": "name"}` — resolved from `self.flags[name].value` at call time
- `{"now": true}` — resolved to `datetime.now()` at call time

## Implementation

### Step 0: Normalize flags format in GET /api/toolkits endpoint (prerequisite)

**File:** `api/routers/toolkits.py` (or the `_build_toolkit_row` helper)

The `flags` JSON column stores two incompatible shapes depending on toolkit origin:
- HANDSHAKE-origin: `{"flag_name": {"type": {"class_name": "Counter_Tracker", ...}, "name": "...", "initial_value": ...}}`
- Backend-authored: `{"flag_name": {"tracker_type": "Counter_Tracker", "initial_value": ...}}`

The UI needs a single consistent shape. Normalize in `_build_toolkit_row` before returning:

```python
def _normalize_flags(flags: dict | None) -> dict:
    if not flags:
        return {}
    normalized = {}
    for name, info in flags.items():
        if "tracker_type" in info:
            tracker_type = info["tracker_type"]
        else:
            type_field = info.get("type", {})
            tracker_type = type_field.get("class_name", "Counter_Tracker") if isinstance(type_field, dict) else "Counter_Tracker"
        normalized[name] = {"tracker_type": tracker_type, "initial_value": info.get("initial_value", 0)}
    return normalized
```

Call this in `_build_toolkit_row` so every `GET /api/toolkits` and `GET /api/toolkits/{id}` response has `flags` as `{flag_name: {tracker_type: string, initial_value}}`.

### Step 2: Backward compat — legacy vs backend-authored toolkit path

The ActionEditor has two paths depending on the toolkit type:

**Backend-authored toolkit (`is_backend_authored=true`):**
- `type: hardware` actions use the **hardware module picker** — dropdown from toolkit's `hardware_module_ids`, method from `GET /api/hardware-modules/{id}/methods`
- This replaces the legacy semantic_hardware key approach

**Legacy toolkit (`is_backend_authored=false` or absent):**
- `type: hardware` actions continue to use the **semantic_hardware dropdown** — the existing `SemanticHardwarePicker` component (keys from toolkit's `semantic_hardware` dict)

### Step 3: Full action editor in StateBodyPanel

**File:** `web_ui/react-src/src/pages/task-editor/TaskEditor.tsx`

The `StateBodyPanel` shows the list of entry_actions for the selected state. Each action has a `type` selector and type-specific fields. Cover all six types:

#### `type: hardware` (backend-authored path only)
1. **Module dropdown** — from toolkit's `hardware_module_ids` list (by name); fetch module names via `GET /api/hardware-modules`
2. **Method dropdown** — `GET /api/hardware-modules/{id}/methods` on module select; cache per module ID in component state
3. **Arg inputs** — one per arg (excluding `__init__`), each supports three modes via toggle: **literal** | **param ref** | **flag ref**
   - Show arg name + annotation as label; pre-fill default if present

Serializes to:
```json
{"type": "hardware", "ref": "Left_LED", "method": "pulse", "args": [{"param": "open_duration"}]}
```

#### `type: flag`
1. **Flag dropdown** — from toolkit's `flags` dict (names + tracker_type from `ToolkitFlag`)
2. **Method dropdown** — expose all methods implemented by the tracker type:
   - `Counter_Tracker`: `increment`, `decrement`, `reset`, `set` (int arg required for set)
   - `Boolean_Tracker`: `set` (bool arg), `toggle` (no arg)
   - `Trial_Tracker`: `increment` (no arg), `set` (int arg)
   - `Tracker` (base): `increment`, `set`
3. **Value input** (for `set`): literal | param-ref | flag-ref modes; type-constrained by tracker_type

`Trial_Tracker.increment()` dispatches `key='INC_TRIAL_COUNTER'` to the orchestrator internally — no special handling needed in the UI. The state builder treats it identically to `Counter_Tracker.increment()`.

Serializes to:
```json
{"type": "flag", "ref": "hit", "method": "increment"}
{"type": "flag", "ref": "rewarded_trial", "method": "toggle"}
```

#### `type: timer`
1. **Timer module dropdown** — from toolkit's hardware modules filtered to timer modules
   - **Timer module identification:** a hardware module is a timer if the linked lib's `filename` is `"timer.py"` OR the module's `class_name` is in a known timer class list (e.g., `["Timer", "CountdownTimer"]`). Check `lib.filename == "timer.py"` as the primary signal.
2. **Method dropdown** — `start`, `stop`, `reset`, `set`
3. **Duration input** (for `start`/`set`): literal | param-ref modes

Serializes to:
```json
{"type": "timer", "ref": "TIMER_TO_REWARD", "method": "set", "args": [{"param": "reward_delay_ms"}]}
```

#### `type: method`
1. **Method name dropdown** — from toolkit's `locked_states` / `selected_states` list
2. **Arg inputs** — freeform (no AST for these — they're Pi task class methods); literal mode only

Serializes to:
```json
{"type": "method", "ref": "prepare_stimulus", "args": []}
```

#### `type: if`
1. **Condition builder** — reuse the same `ConditionBuilder` component used for transition conditions:
   - Left operand: dropdown — flag | param | view | literal
   - Operator: `==`, `!=`, `>=`, `<=`, `>`, `<`
   - Right operand: same dropdown
   - **`ConditionBuilder` must use `getParamKeys(toolkit)` for param dropdown** — `Object.keys(toolkit?.params_schema)` breaks when params_schema is an array (backend-authored toolkits). Pass the same `getParamKeys` helper used in `ArgInput`.
   - Flag dropdown populates from `Object.keys(toolkit?.flags ?? {})` (works after Step 0 normalization)
   - Both dropdowns must be visible and selectable in the rendered card — the user must be able to pick flag/param/literal on both sides without typing freehand
2. **then-branch**: nested action list (recursive, same action editor, supports all types)
3. **else-branch**: optional nested action list

Serializes to:
```json
{
  "type": "if",
  "condition": {"left": {"flag": "hit"}, "op": ">=", "right": {"param": "hit_threshold"}},
  "then": [{"type": "hardware", "ref": "Valve_1", "method": "pulse", "args": [{"param": "reward_ms"}]}],
  "else": []
}
```

Implementation notes:
- Cache hardware module methods per module ID in component state to avoid repeated fetches
- The `ref` field for hardware/flag/timer must match the exact `name` field (not display_name or id)
- `ConditionBuilder` should already exist from transition editing — reuse it for `type: if`
- `type: if` branches are recursively rendered using the same action editor component

## Step 5: UI Design System (non-technical UX)

The editor must be usable by researchers with no coding ability. All controls must auto-suggest valid options wherever possible; every label must have a hover tooltip explaining what it does in plain English.

### Color accent per action type

Each action card and related pill/badge is tinted with a consistent color accent so users can scan the action list at a glance:

| Type | Accent color | Hex | Used for |
|---|---|---|---|
| `hardware` | Blue | `#3b82f6` | Card left-border, type chip, active module/method highlight |
| `flag` | Amber | `#f59e0b` | Card left-border, type chip, flag-mode arg pill |
| `timer` | Teal | `#14b8a6` | Card left-border, type chip |
| `method` | Lavender | `#a78bfa` | Card left-border, type chip |
| `if` | Green | `#22c55e` | Card left-border, type chip, then/else branch markers |

Card background is `rgba(<accent-rgb>, 0.06)`, left-border is `3px solid <accent>`, outer border is `1px solid rgba(<accent-rgb>, 0.25)`.

In **ArgInput** the source-of-value mode indicator also has semantic colors:
- **Literal** (`#`): neutral grey — a fixed, hard-coded value
- **Param** (`$`): green `#22c55e` — resolved from protocol parameters at runtime
- **Flag** (`!`): amber `#f59e0b` — resolved from a flag's current value at runtime

### Tooltip spec

Every label with `ⓘ` suffix gets a native `title` attribute. Required tooltips:

| Control | Tooltip text |
|---|---|
| Type chip / type selector | "What kind of action executes when this state is entered?" |
| Device (hardware module) | "Which physical device should respond? Devices come from the toolkit's hardware module list." |
| Action (hardware method) | "What the device should do. Options come from the hardware library's Python class methods." |
| Arg value (each arg) | The arg annotation from AST, e.g. "float — duration in seconds. Default: 0.1" |
| Flag selector | "Which counter, boolean, or trial marker to update?" |
| Operation (flag method) | The per-method description (see table below) |
| Timer selector | "Which timer device to control?" |
| Operation (timer method) | The per-method description |
| ArgInput mode: Literal | "A fixed value baked into the FDA. Does not change between runs." |
| ArgInput mode: Param ($) | "Resolved from a protocol parameter at runtime. Set in the protocol step config." |
| ArgInput mode: Flag (!) | "Resolved from a flag's current value at runtime. Changes during the session." |

Flag operation descriptions (shown in dropdown `title` and inline hint):

| method | description |
|---|---|
| `increment` | Add 1 to this counter |
| `decrement` | Subtract 1 from this counter |
| `reset` | Reset to its starting value |
| `set` | Set to an exact value |
| `toggle` | Flip between true and false |

Timer method descriptions:

| method | description |
|---|---|
| `start` | Start counting down with the given duration |
| `stop` | Stop the timer (does not reset) |
| `reset` | Reset the timer to zero |
| `set` | Set the duration without starting |

### Auto-fill behavior

- **On type change**: auto-select the first valid option for every new field (first module, first method for that module, first param for arg, etc.)
- **On flag change**: auto-update the method dropdown to the first method valid for the new flag's tracker type
- **On module change (hardware)**: clear method and args; immediately fetch methods for the new module; auto-select first method when loaded
- **Method dropdown disabled with loading text** ("Loading…") while `/api/hardware-modules/{id}/methods` is in flight
- Hardware module methods are **cached in a module-level JS object** (`const METHOD_CACHE: Record<number, AstMethod[]> = {}`) so repeated selections within the same session don't re-fetch

### ArgInput improvements

Current: only supports `number` literals. Required:
- Mode buttons are pill-shaped tabs with a color accent when active:
  - `#` Literal → grey when active (`#6b7280`)
  - `$` Param → green when active (`#22c55e`), select shows param names
  - `!` Flag → amber when active (`#f59e0b`), select shows flag names
- Param/flag dropdowns show the symbolic prefix in the selected display: `$open_duration`, `!hit`
- `params_schema` may be an array `[{name, type, default}]` (backend-authored) or a dict `{name: {type, default}}` (HANDSHAKE). A helper `getParamKeys(toolkit)` must handle both shapes.

#### Arg type inference from AST annotation (no extra metadata needed)

`AstMethodArg.annotation` is already the Python type annotation string (e.g. `"float"`, `"int"`, `"bool"`, `"str"`, `"Optional[float]"`). Use a helper to map it to an input kind:

```typescript
function annotationToInputKind(ann: string | null): 'number' | 'bool' | 'text' {
  if (!ann) return 'text'
  const base = ann.replace(/Optional\[|\]/g, '').trim()
  if (base === 'bool') return 'bool'
  if (base === 'int' || base === 'float') return 'number'
  return 'text'  // str, complex types, unannotated → text fallback
}
```

Literal input per kind:
- `number` → `<input type="number" step="any">` — shows numeric keyboard, accepts decimals
- `bool` → `<select>` with `true` / `false` options — serialized as JSON boolean
- `text` → `<input type="text">` — plain string, covers unannotated args and str types

**No need to add extra typing to hardware libs.** The Python annotation covers all common hardware method args (durations as `float`, counts as `int`, pin states as `bool`). Args with no annotation fall back to text, which is acceptable.

ArgInput receives an optional `annotation: string | null` prop (from the parent ActionEditor's `currentMethod.args[i].annotation`) and uses `annotationToInputKind` to pick the literal input type. Default value from `AstMethodArg.default` is pre-filled if the arg is currently empty.
